# 恭喜完成实验！🎉

你已经在一个真实的 Vault 服务器上**深入观察**了 Shamir 阈值机制，并完整执行了 Rotate 与 Rekey 这两个长得像但作用截然不同的关键命令。

## 本实验的核心收获

| 阶段 | 你亲手验证的事实 |
| :--- | :--- |
| **Nonce 机制** | 每次解封会话有唯一标识，防止并发解封被劫持 |
| **少于 K 份卡住** | Vault 不会"半解封"——少于阈值的提交在数学上零信息泄漏，业务直接 503 |
| **任意 K 份等价** | 第 1+2+4 份和第 1+2+3 份完全一样，提供单点缺勤容错 |
| **`unseal -reset`** | 一键中止当前解封会话，旧 Nonce 失效 |
| **Rotate** | DEK 轮转，无需分片授权，老数据 keyring 兼容 |
| **Rekey** | Unseal Key 重生 + 分片重切，必须用当前阈值授权，老分片立即失效 |
| **吊销初始 Root** | 生产环境的标准收尾动作 |

## 心智模型：何时用什么命令

| 你想做什么 | 用什么 |
| :--- | :--- |
| 临时停掉 Vault 用于安全事件响应 | `vault operator seal` |
| 紧急情况后恢复服务 | 收集 K 份分片 → `vault operator unseal` × K |
| 我提交的分片可能错了，想从头来 | `vault operator unseal -reset` |
| 定期轮换加密密钥 | `vault operator rotate` |
| 持有分片的人员变动 | `vault operator rekey -init` → 收集 K 份当前分片 → 分发新分片 |
| 初始化后清理高权限 Token | `vault token revoke <root-token>` |
| 后续需要管理权限 | `vault operator generate-root`（需分片授权） |

## Rotate vs Rekey：必须刻在脑子里的区别

```
┌─────────────────────────────────────────────────────────────┐
│  rotate                                                      │
│  ─ 作用层：DEK（最内层）                                     │
│  ─ 谁能执行：已解封的管理员（无需分片）                      │
│  ─ 频率：高（月度自动化）                                    │
│  ─ 影响范围：未来加密用新密钥；老数据 keyring 兼容           │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  rekey                                                       │
│  ─ 作用层：Unseal Key（最外层）                              │
│  ─ 谁能执行：必须用 ≥ 当前 threshold 的分片授权              │
│  ─ 频率：低（人员变动时触发）                                │
│  ─ 影响范围：老分片立即失效；下次解封必须用新分片            │
└─────────────────────────────────────────────────────────────┘
```
