# 2.1 "Dev" 开发模式的适用边界与安全风险预警

> **核心结论**：`vault server -dev` 是一把锋利的双刃剑——它让你在 30 秒内拥有一个全功能的 Vault 实例，但它的每一个"方便之处"背后都是一个具体、可被利用的安全缺陷。本节的目标是让你彻底理解这些缺陷，并在脑海里刻下一条不可逾越的红线。

## 1. Dev 模式的七大特性

根据官方文档 [Dev server mode](https://developer.hashicorp.com/vault/docs/concepts/dev-server)，当你运行 `vault server -dev` 时，Vault 会短路（short-circuit）大量正常的安全配置，以换取零配置启动体验：

| 特性 | Dev 模式的行为 | 对应的安全含义 |
| :--- | :--- | :--- |
| **初始化状态** | 自动完成 `vault operator init` | 无需人工干预，但也跳过了 Shamir 密钥分片 |
| **封印状态** | 自动解封（Unsealed） | 启动即可用，但绕过了"多人持有解封密钥"的物理安全机制 |
| **存储后端** | **纯内存存储**（in-memory） | 重启即丢失所有数据，包括策略、密钥、凭据 |
| **TLS** | **禁用**，监听 `http://127.0.0.1:8200` | 网络流量明文传输，任何能嗅探网络的人都能截获 Token 和机密 |
| **Root Token** | 默认为已知的固定值（`hvs.root` 或通过 `-dev-root-token-id` 指定） | 任何知道此 Token 的人都拥有 Vault 的完整管理权限 |
| **Unseal Key** | 只有 1 个分片（1/1 Shamir）且被打印在终端 | 控制台访问权 = Vault 完整控制权 |
| **预挂载引擎** | KV v2 自动挂载于 `secret/` | 可立即使用，但所有历史数据在重启时蒸发 |

## 2. 具体安全风险的深度解析

### 2.1 内存存储：数据生命周期的幻觉

Dev 模式使用的 `inmem` 存储后端将所有 Vault 数据（包括机密、策略、认证方法配置、租约记录）存储在进程内存中。这带来了以下直接后果：

- **进程崩溃 = 数据永久消失**：不同于生产模式下有 WAL（Write-Ahead Log）和 Raft 日志可以恢复，内存数据随进程终止而消失，没有任何恢复路径。
- **没有 ACID 保证**：生产环境下 Vault 依赖存储后端的持久化来保证事务一致性；内存模式下这一保证形同虚设。
- **误导开发者**：开发者在 Dev 模式下测试"动态数据库凭据"看起来很完美，但如果存储后端配置不当，生产环境的租约续期逻辑可能表现完全不同。

### 2.2 TLS 禁用：明文传输的巨大攻击面

当 `VAULT_ADDR=http://127.0.0.1:8200` 时，所有与 Vault 的通信都通过未加密的 HTTP 进行。这意味着：

- **Token 在每个 HTTP 请求的 Header 中以明文传输**：`X-Vault-Token: root`。任何能读取网络流量的攻击者（包括同一台机器上的其他进程、同一局域网的监听者）都能截获这个 Token。
- **响应体中的机密同样明文暴露**：当你读取一个 KV 机密时，响应 JSON 里的 `value` 字段以明文出现在网络包中。
- **合规性破坏**：SOC2、PCI-DSS 等合规框架的任何审计都会立即标记无 TLS 的机密管理系统为严重违规。

> **注意**：Dev 模式提供了 `-dev-tls` 参数，可以自动生成自签名证书并启用 HTTPS，但这仅比裸 HTTP 好一点——自签名证书无法通过公钥基础设施验证，仍不适用于生产。

### 2.3 可预测/公开的 Root Token：最高权限的平民化

在 Dev 模式下，Root Token 要么是默认的 `hvs.root`（或旧版的字符串 `root`），要么是通过命令行 `-dev-root-token-id` 明文指定的。无论哪种情况：

```bash
# 任何能运行这条命令的人都能看到 Root Token
ps aux | grep "vault server"
# 输出：... vault server -dev -dev-root-token-id=root ...
```

Root Token 拥有 Vault 内置的 `root` 策略，等同于无限制的超级管理员。在 Dev 模式中，这本质上是**将数据库 root 密码贴在终端屏幕上**。

### 2.4 单一解封密钥：Shamir 机制的完全绕过

生产环境中，Shamir's Secret Sharing 将解封密钥分拆成 N 份，需要 K 人各自提供一份才能解封（默认 5/3）。这是一种强制的"多人授权"机制，防止任何单一人员拥有完全控制权。

Dev 模式只有 1 个分片（threshold = 1, shares = 1），且这个分片被打印在终端输出里。这彻底废除了 Shamir 机制本来要保障的"多人控制"安全原则。

### 2.5 监听地址的横向暴露风险

默认情况下 Dev 模式监听 `127.0.0.1:8200`（仅本地回环），看似安全。但开发者常常为了从容器或远程 IDE 访问而主动修改：

```bash
# 危险！这会将无 TLS 的 Vault 暴露到所有网络接口
vault server -dev -dev-listen-address="0.0.0.0:8200"
```

一旦监听在 `0.0.0.0`，同一局域网（或云上同一 VPC）内的任何主机都能用 Root Token `root` 获得完整控制权。

## 3. 适用边界：Dev 模式的正确使用场合

基于以上分析，可以建立明确的使用边界决策表：

| 使用场景 | Dev 模式是否适用 | 原因 |
| :--- | :---: | :--- |
| **本教程的动手实验** | ✅ | 实验目的是学习 API 用法，不涉及真实数据 |
| **本地开发机：30 分钟功能验证** | ✅ | 短生命周期，数据无需持久化，无人监听本地流量 |
| **CI/CD 流水线单元测试** | ✅ | 短暂、隔离的测试容器，测试结束即销毁 |
| **团队共享的开发服务器** | ❌ | 多人共用环境，Root Token 公开，风险极高 |
| **Staging / 预发布环境** | ❌ | Staging 应尽量模拟生产，Dev 模式的行为差异会掩盖真实问题 |
| **任何持久化数据的场景** | ❌ | 内存存储，重启即丢失 |
| **生产环境** | ❌🚫 | 永远不允许，任何情况不例外 |
| **Docker Compose 集成测试（对外暴露端口）** | ❌ | 若容器端口映射到宿主机，等同于公开 Root Token |

## 4. Dev 模式 vs 生产模式：一键对比

```
vault server -dev                    vault server -config=/etc/vault.d/vault.hcl
│                                    │
├─ 存储：内存（重启丢失）           ├─ 存储：Integrated Storage (Raft)
├─ TLS：无（http://）               ├─ TLS：开启（https://，证书由 PKI 签发）
├─ 解封：自动（绕过 Shamir）        ├─ 解封：手动 Shamir 或 Auto-Unseal (KMS)
├─ Root Token：已知固定值           ├─ Root Token：初始化时随机生成，用完即销毁
├─ 初始化：自动                     ├─ 初始化：`vault operator init`（明确操作）
├─ 审计：不启用                     ├─ 审计：强制开启 File/Syslog 设备
└─ 适用：本地学习 / CI 测试         └─ 适用：一切真实数据场景
```
